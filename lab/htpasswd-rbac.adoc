= Configuring htpasswd Authentication and RBAC

OpenShift supports a number of different authentication providers. One of the
simplest to configure for workshops and smaller environments is the htpasswd
identity provider. In this lab, you will configure htpasswd authentication,
create users, and then implement Role-Based Access Control (RBAC) to manage
what those users can do.

By the end of this lab you will understand:

* How to configure the htpasswd identity provider
* How OpenShift *Users* and *Groups* work
* How to assign *Roles* and *ClusterRoles* to users and groups
* How to verify RBAC policies by testing as different users

=== Background: OAuth in OpenShift

OpenShift uses an internal OAuth server to manage authentication. The OAuth
server can be configured with one or more identity providers. In a fresh
installation, only the `kubeadmin` user exists, backed by a *Secret* in the
`kube-system` namespace.

==== Examine the current OAuth configuration

Make sure you are logged in as `kubeadmin`:

[source,bash,role="execute"]
----
oc login -u kubeadmin -p {{ KUBEADMIN_PASSWORD }}
----

Examine the current OAuth configuration:

[source,bash,role="execute"]
----
oc get oauth cluster -o yaml
----

You will see something like:

[source,yaml]
----
apiVersion: config.openshift.io/v1
kind: OAuth
metadata:
  name: cluster
spec: {}
----

The `spec` is empty — the only authentication is the `kubeadmin` user, which
is backed by a *Secret*. You can verify this:

[source,bash,role="execute"]
----
oc get secret -n kube-system kubeadmin
----

[Note]
====
The `kubeadmin` account will continue to work even after you configure a new
identity provider. To fully disable it, you would delete the `kubeadmin`
*Secret* from the `kube-system` namespace. **Do not do this during this lab.**
====

=== Creating Users with htpasswd

The htpasswd identity provider uses a flat file containing usernames and
hashed passwords. We will create several users to simulate different roles in
an organization:

* `admin-user` — a platform administrator
* `dev-user1` — a developer on Team Alpha
* `dev-user2` — a developer on Team Alpha
* `viewer-user` — an auditor with read-only access

==== Install the htpasswd utility

The `htpasswd` command is part of the `httpd-tools` package. It may already be
installed in your environment. Verify with:

[source,bash,role="execute"]
----
which htpasswd || echo "htpasswd not found"
----

If it is not installed, you can install it with:

[source,bash,role="execute"]
----
sudo dnf install -y httpd-tools
----

==== Create the htpasswd file

Create a new htpasswd file with the first user, then add additional users:

[source,bash,role="execute"]
----
htpasswd -c -B -b /tmp/htpasswd admin-user openshift123
htpasswd -B -b /tmp/htpasswd dev-user1 openshift123
htpasswd -B -b /tmp/htpasswd dev-user2 openshift123
htpasswd -B -b /tmp/htpasswd viewer-user openshift123
----

[Note]
====
The `-c` flag creates the file (only used for the first user). The `-B` flag
uses bcrypt hashing, which is the recommended algorithm. The `-b` flag takes
the password from the command line.
====

Verify the file was created with all four users:

[source,bash,role="execute"]
----
cat /tmp/htpasswd
----

You should see four lines, each with a username and a bcrypt-hashed password.

=== Configuring the OAuth Provider

To configure OpenShift to use the htpasswd file, you must:

1. Create a *Secret* in the `openshift-config` namespace containing the htpasswd file
2. Update the `cluster` *OAuth* resource to reference the htpasswd identity provider

==== Create the Secret

[source,bash,role="execute"]
----
oc create secret generic htpasswd-secret --from-file=htpasswd=/tmp/htpasswd -n openshift-config
----

==== Update the OAuth configuration

Apply the htpasswd identity provider configuration:

[source,bash,role="execute"]
----
oc apply -f - <<EOF
apiVersion: config.openshift.io/v1
kind: OAuth
metadata:
  name: cluster
spec:
  identityProviders:
  - name: htpasswd
    challenge: true
    login: true
    mappingMethod: claim
    type: HTPasswd
    htpasswd:
      fileData:
        name: htpasswd-secret
EOF
----

[Note]
====
We use `apply` because the `OAuth` object already exists. If you used `create`
you would get an error.
====

This triggers a redeployment of the OAuth pods. Wait for the rollout to
complete:

[source,bash,role="execute"]
----
oc rollout status deployment/oauth-openshift -n openshift-authentication
----

[Note]
====
This may take a minute or two. If the rollout seems stuck, wait 30 seconds
and run the command again. You can also monitor with:
`oc get pods -n openshift-authentication -w`
====

==== Test the new users

Try logging in as each user to verify the htpasswd provider is working. If
you get a login error, wait a minute for the OAuth pods to finish rolling out
and try again.

[source,bash,role="execute"]
----
oc login -u admin-user -p openshift123
----

You should see:

----
Login successful.

You don't have any projects. You can try to create a new project, by running

    oc new-project <projectname>
----

Verify the user:

[source,bash,role="execute"]
----
oc whoami
----

Now check that the *User* object was created:

[source,bash,role="execute"]
----
oc login -u kubeadmin -p {{ KUBEADMIN_PASSWORD }}
oc get users
----

You should see:

----
NAME         UID                                    FULL NAME   IDENTITIES
admin-user   xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx               htpasswd:admin-user
----

[Note]
====
*Users* are only created after their first login. You will not see the other
users until they log in for the first time.
====

Let's log in as the remaining users so their *User* objects are created:

[source,bash,role="execute"]
----
oc login -u dev-user1 -p openshift123
oc login -u dev-user2 -p openshift123
oc login -u viewer-user -p openshift123
----

Switch back to `kubeadmin` and verify all users exist:

[source,bash,role="execute"]
----
oc login -u kubeadmin -p {{ KUBEADMIN_PASSWORD }}
oc get users
----

You should now see all four users listed.

=== Creating Groups

*Groups* allow you to manage permissions for multiple users at once. Unlike
LDAP-synced groups, we will create these manually since htpasswd does not have
a group concept.

==== Create the groups

[source,bash,role="execute"]
----
oc adm groups new platform-admins admin-user
oc adm groups new developers dev-user1 dev-user2
oc adm groups new auditors viewer-user
----

Verify the groups were created:

[source,bash,role="execute"]
----
oc get groups
----

You should see:

----
NAME              USERS
platform-admins   admin-user
developers        dev-user1, dev-user2
auditors          viewer-user
----

Examine a group in detail:

[source,bash,role="execute"]
----
oc get group developers -o yaml
----

The YAML looks like:

[source,yaml]
----
apiVersion: user.openshift.io/v1
kind: Group
metadata:
  name: developers
users:
- dev-user1
- dev-user2
----

=== Implementing RBAC

Now that we have users and groups, we can assign roles to control what each
group can do. OpenShift comes with several built-in roles:

* `cluster-admin` — super-user, full access to everything
* `cluster-reader` — read-only access to all cluster resources
* `admin` — project-level admin (view/modify most resources except quotas)
* `edit` — modify most objects in a project
* `view` — read-only access to a project

==== Grant cluster-admin to platform-admins

The `platform-admins` group should have full administrative access to the
cluster:

[source,bash,role="execute"]
----
oc adm policy add-cluster-role-to-group cluster-admin platform-admins
----

Verify by logging in as `admin-user`:

[source,bash,role="execute"]
----
oc login -u admin-user -p openshift123
oc get nodes
----

You should see the list of cluster nodes — confirming cluster-admin access.

[source,bash,role="execute"]
----
oc get projects | head -10
----

You should see all projects in the cluster.

==== Grant cluster-reader to auditors

Auditors need to see everything but modify nothing:

[source,bash,role="execute"]
----
oc login -u kubeadmin -p {{ KUBEADMIN_PASSWORD }}
oc adm policy add-cluster-role-to-group cluster-reader auditors
----

Test as the `viewer-user`:

[source,bash,role="execute"]
----
oc login -u viewer-user -p openshift123
oc get projects | head -10
----

You should see all projects listed. Now try to create something:

[source,bash,role="execute"]
----
oc new-project auditor-test
----

You should see an error or the project is created (because `self-provisioner`
is still active). Let's test a more restricted action — try to delete a system
namespace:

[source,bash,role="execute"]
----
oc delete project openshift-console
----

You should see:

----
Error from server (Forbidden): projects.project.openshift.io "openshift-console" is forbidden: User "viewer-user" cannot delete resource "projects" in API group "project.openshift.io" in the namespace "openshift-console"
----

The `cluster-reader` role allows viewing but not modifying cluster resources.

==== Create projects for developers

Switch back to `kubeadmin` and create projects that represent a typical
development workflow:

[source,bash,role="execute"]
----
oc login -u kubeadmin -p {{ KUBEADMIN_PASSWORD }}
oc adm new-project team-alpha-dev --display-name="Team Alpha - Development"
oc adm new-project team-alpha-stage --display-name="Team Alpha - Staging"
oc adm new-project team-alpha-prod --display-name="Team Alpha - Production"
----

==== Assign project-level roles to developers

Give the `developers` group edit access to dev and staging, but only view
access to production:

[source,bash,role="execute"]
----
oc adm policy add-role-to-group edit developers -n team-alpha-dev
oc adm policy add-role-to-group edit developers -n team-alpha-stage
oc adm policy add-role-to-group view developers -n team-alpha-prod
----

==== Test developer access

Log in as `dev-user1` and see what projects are visible:

[source,bash,role="execute"]
----
oc login -u dev-user1 -p openshift123
oc get projects
----

You should see the three `team-alpha-*` projects (and any self-provisioned
projects).

Try deploying an application to the dev project:

[source,bash,role="execute"]
----
oc project team-alpha-dev
oc create deployment hello --image=quay.io/redhatworkshops/welcome-php:latest
oc rollout status deployment hello
----

You should see the deployment succeed. Now verify the pod is running:

[source,bash,role="execute"]
----
oc get pods
----

Now try to deploy to the production project:

[source,bash,role="execute"]
----
oc project team-alpha-prod
oc create deployment hello-prod --image=quay.io/redhatworkshops/welcome-php:latest
----

You should see an error:

----
error: failed to create deployment: deployments.apps is forbidden: User "dev-user1" cannot create resource "deployments" in API group "apps" in the namespace "team-alpha-prod"
----

This is exactly what we want — developers can deploy to dev and staging, but
can only **view** production.

Verify view access works in production:

[source,bash,role="execute"]
----
oc get pods -n team-alpha-prod
----

This should succeed (even if there are no pods to list) — read access is
granted.

==== Examine RoleBindings

Let's inspect what we've created. Switch back to `kubeadmin`:

[source,bash,role="execute"]
----
oc login -u kubeadmin -p {{ KUBEADMIN_PASSWORD }}
----

Look at the RoleBindings in the dev project:

[source,bash,role="execute"]
----
oc get rolebindings -n team-alpha-dev
----

You will see several default bindings plus the one we created. Find the
`edit` binding for our developers group:

[source,bash,role="execute"]
----
oc get rolebinding -n team-alpha-dev -o wide | grep developers
----

Examine the details:

[source,bash,role="execute"]
----
oc describe rolebinding -n team-alpha-dev edit
----

You should see output showing the `edit` *ClusterRole* bound to the
`developers` *Group* in the `team-alpha-dev` namespace.

Now compare with the ClusterRoleBindings we created:

[source,bash,role="execute"]
----
oc get clusterrolebindings | grep -E "platform-admins|auditors"
----

You should see bindings for both groups:

----
cluster-admin-0    ClusterRole/cluster-admin    ...
cluster-reader-0   ClusterRole/cluster-reader   ...
----

[Note]
====
The key difference:

* A *RoleBinding* grants access within a specific namespace (project)
* A *ClusterRoleBinding* grants access across the entire cluster

Both can reference a *ClusterRole*, but a *RoleBinding* limits its scope to
one namespace.
====

=== Adding a User to an Existing Group

What if a new developer joins Team Alpha? You can add them to the `developers`
group, and they will automatically inherit all the RoleBindings assigned to
that group.

First, let's add a new user to the htpasswd file and update the secret:

[source,bash,role="execute"]
----
oc get secret htpasswd-secret -n openshift-config -o jsonpath='{.data.htpasswd}' | base64 -d > /tmp/htpasswd
htpasswd -B -b /tmp/htpasswd dev-user3 openshift123
oc set data secret/htpasswd-secret -n openshift-config --from-file=htpasswd=/tmp/htpasswd
----

[Note]
====
When you update the htpasswd *Secret*, the OAuth operator detects the change
and triggers a rollout automatically. Wait for it to complete before
proceeding.
====

[source,bash,role="execute"]
----
sleep 15
oc rollout status deployment/oauth-openshift -n openshift-authentication
----

Now log in as the new user to create the *User* object, then add them to the
group:

[source,bash,role="execute"]
----
oc login -u dev-user3 -p openshift123
oc login -u kubeadmin -p {{ KUBEADMIN_PASSWORD }}
oc adm groups add-users developers dev-user3
----

Verify:

[source,bash,role="execute"]
----
oc get group developers
----

You should see:

----
NAME         USERS
developers   dev-user1, dev-user2, dev-user3
----

Test that `dev-user3` automatically inherits the same access:

[source,bash,role="execute"]
----
oc login -u dev-user3 -p openshift123
oc project team-alpha-dev
oc get pods
----

The new user has edit access to the dev project without any additional
RoleBinding configuration — that's the power of group-based RBAC.

=== Summary

In this lab you have:

* Configured the htpasswd identity provider on OpenShift
* Created users and groups
* Assigned cluster-level roles (`cluster-admin`, `cluster-reader`)
* Assigned project-level roles (`edit`, `view`) to groups
* Tested that RBAC policies correctly allow and deny access
* Added a new user to an existing group and verified inherited permissions

The key takeaways:

* *Roles* define what actions are allowed (rules)
* *RoleBindings* and *ClusterRoleBindings* attach roles to users or groups
* *Groups* make RBAC manageable — assign roles to groups, not individual users
* If a user is not explicitly granted access, they are **denied** by default

[source,bash,role="execute"]
----

----
